# 01 · 产品愿景

> [English version](../01-product-vision.md) · [ai-gateway 文档套件](README.md)的一部分

## 一句话定位

**ai-gateway 是一个自托管的 AI 流量控制平面：一个 Go 单二进制，把虚拟 Key、配额、审计、计费和多供应商路由放在你的应用与所有大模型之间。**

"控制平面"这个词是刻意选择的。代理只是转发流量；控制平面治理流量。这个项目的价值不在于中转 HTTP 请求，而在于中转之外发生的一切：谁可以调用哪个模型、可以花多少钱、实际消耗了什么、成本几何、发送和返回了什么内容，以及某个提供方劣化时会发生什么。

## 项目为什么存在

采用大模型的组织会以同样的顺序撞上同一堵墙：

1. **第 1 周** —— 开发者把提供方 API Key 直接嵌进应用。能跑。
2. **第 1 个月** —— Key 泄漏进代码仓库，没人知道哪个团队花了多少钱，切换提供方意味着改动每一个应用。
3. **第 3 个月** —— 财务要求按团队做成本归集，安全要求对发送给外部模型的内容留有审计痕迹，工程要求自动故障转移——因为一次单一供应商故障同时打挂了所有 AI 功能。

每一个认真使用大模型的组织最终都会自建一个内部网关。ai-gateway 就是那个网关：造一次、造好、开源。

## 目标用户

三类用户画像驱动全部需求。每个路线图条目至少要服务其中之一；三者都不服务的功能不做。

### 1. 平台团队（企业内部成本管控）

公司内多个团队消费大模型，平台/基础设施团队负责治理。他们需要：

- 按团队/项目发放虚拟 Key，各自独立的配额与消费上限
- 成本归集与分摊报表（"A 团队上月在 gpt-4o 上花了 4 万元"）
- 合规级的提示词与补全内容审计（含 PII 管控）
- 提供方故障转移，避免单一供应商故障级联
- 必须自托管：提示词不能离开自己的网络边界

### 2. API 转售商 / 聚合服务商

从上游购买额度、加价转售给自己客户的运营者。他们需要平台团队的一切，另加：

- 预付费余额账户：充值、扣减、欠费停用
- 面向客户的售价与上游成本解耦（阶梯价、分组折扣）
- 订阅套餐、支付网关对接、发票
- 客户之间的硬多租户隔离

### 3. SaaS 产品团队（内嵌 LLM 能力）

在自己的 SaaS 中集成 LLM 功能的产品团队。他们需要：

- 通过短期虚拟 Key 做终端用户级限流与滥用防护
- 模型映射：产品侧引用 `our-summarizer-v2`，后端自由替换真实模型
- 会话亲和以获得提示词缓存收益
- 延迟优先的路由与语义缓存，控制单位经济性

### 空位在哪里

逐列读矩阵，能看到这个项目要填补的缺口：

- **LiteLLM** 是覆盖最广的协议翻译器，但它是 Python（高并发下运维更重），且止步于预算——没有商业计费闭环。
- **One API / New API** 在中文生态验证了"Go 单二进制 + 余额制转售"的需求，但审计、护栏、可观测性与工程质量（测试、文档）薄弱。
- **Portkey / Helicone** 很优秀但 SaaS 优先；完全自托管是二等公民。
- **Kong / Higress** 在 L7 是网关级选手，但把 AI 当作插件；虚拟 Key 经济体系、计费和 LLM 原生审计不是它们的重心。

**尚未被占据的位置：自托管 Go 单二进制，同时具备网关级流量治理、完整商业计费闭环与合规级审计。** 这就是 ai-gateway 瞄准的位置。画像 2（转售商）今天没有一等公民的开源选项；画像 1 和 3 目前必须拼装 2–3 个工具才能覆盖一个网关应该做完的事。

## 设计原则

这些原则用来裁决日常设计分歧。两个方案冲突时，满足更多原则的胜出。

1. **OpenAI 兼容优先。** `/ai/v1/*` 是对外契约。任何讲 OpenAI 协议的客户端必须零改动可用。原生协议（Anthropic、Gemini）是增量入口，永远不是替代。
2. **单二进制、最少依赖。** `./server -conf config.yaml` 加上 MySQL/PostgreSQL 与 Redis 就是一套完整生产部署。Elasticsearch、向量库、支付网关严格可选，缺席时优雅降级。
3. **无状态、可水平扩展。** 共享状态全部放在数据库与 Redis。任何实例可服务任何请求；杀掉一个实例不丢任何东西。（现有的 L1 缓存 + Redis pub/sub 失效模式，见 `internal/biz/key_cache.go`，就是模板。）
4. **Headless 优先。** 每个能力先是 API，然后才是界面。Web 控制台只是管理 API 的一个客户端，零私有接口。
5. **热路径神圣不可侵犯。** 代理路径的附加延迟预算是个位数毫秒。任何更重的事情——审计落库、计费结算、PII 深度扫描——都异步化（现有 `AuditWorker` 的批处理模式就是模板）。
6. **经济问题失败开放，安全问题失败关闭。** 计费或配额子系统劣化时，由可配置策略决定放行与否（默认放行、事后对账）。认证或阻断型护栏劣化时，流量停止。
7. **计量必可溯源。** 出现在账单上的每一个数字都必须能追溯到具体的审计日志记录。没有出处的计费是客服灾难。
8. **为还没见过的协议设计。** 适配器、Hook、归一化层都是接口，不是 switch 语句。LLM API 版图每个季度都在变（Responses API、MCP、Agent 工具调用）；架构必须在不重写内核的前提下吸收新形态。

## ai-gateway 不是什么

克制的边界本身就是功能：

- **不是推理服务器。** 它从不运行模型，只治理通往模型的流量（包括自托管的 vLLM/Ollama——它们不过是 OpenAI 兼容提供方）。
- **不是提示词工程平台。** 不做提示词版本、A/B 测试或 playground。可观测数据可导出给做这些事的工具。
- **不是通用 API 网关。** 不打算替代 Kong/Nginx 处理非 AI 流量。已有网关的用户可以把 ai-gateway 放在它后面。
- **不是 Agent 框架。** 它会治理与审计 Agent 流量（P3：MCP 网关），但构建 Agent 属于客户端框架的职责。

## 成功标准（24 个月视野）

- 一个平台团队从 `git clone` 到带故障转移与看板的生产部署，用时不到一小时。
- 一个转售商仅凭本项目即可运营付费 API 业务（充值 → 消费 → 开票）。
- 一名安全负责人仅通过控制台就能回答"上季度我们向外部模型发送了什么、是否涉及 PII"。
- 社区在不改内核代码的情况下贡献新的提供方适配器。
